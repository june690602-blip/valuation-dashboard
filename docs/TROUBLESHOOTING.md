# 트러블슈팅 로그

개발 중 실제로 부딪힌 문제와 해결 과정을 기록한다. 각 항목은
**증상 → 원인 → 해결 → 교훈**으로 정리했다. "왜 그렇게 고쳤는지"를 남겨,
같은 문제를 반복하지 않고 설계 판단의 근거를 추적할 수 있게 하는 것이 목적이다.

---

## 1. 왼쪽 사이드바 고정이 먹지 않음 (인라인 스타일 vs 미디어쿼리)

- **증상**: 주식 상세 화면에서 스크롤을 내리면 시장·종목 입력 사이드바가 본문과 함께
  스크롤되어 사라졌다. 데스크톱 전용 CSS로 `.app-layout { height: calc(100vh - 58px) }`,
  `.app-content { overflow-y: auto }`를 넣었는데도 본문이 뷰포트 안에서 스크롤되지 않고
  페이지 전체가 스크롤됐다.
- **원인**: 두 단계였다.
  1. `.app-layout`은 `min-height:100vh`인 컬럼의 flex 자식이라 `flex:1`(=`flex-grow:1`,
     `min-height:auto`)이면 **콘텐츠 높이만큼 자라나** `height` 지정이 무력화된다
     (자라는 flex 아이템은 `height`로 고정되지 않는다).
  2. `flex:none`으로 바꿔도 소용없었는데, `.app-layout`에 **인라인 `style="flex:1"`**이
     걸려 있어 스타일시트(미디어쿼리) 규칙을 이겼다. 인라인 스타일은 `!important` 없는
     스타일시트 규칙보다 우선한다.
- **해결**: 미디어쿼리에서 `.app-layout { flex: none !important; height: calc(100vh - 58px); }`.
  이제 레이아웃이 뷰포트 높이로 고정되고 `.app-content`만 내부 스크롤 → 헤더·사이드바·
  토글이 모두 제자리에 남는다. 모바일(<901px)은 규칙을 적용하지 않아 일반 페이지 스크롤 유지.
- **교훈**: "안 먹는 CSS"는 대개 (a) flex 성장 규칙, (b) 명시도(인라인/`!important`) 문제다.
  `getComputedStyle`로 실제 적용값을 찍어 두 가지를 먼저 확인한다.

## 2. "3개 방법 평균 적정가" 문구가 계속 보임 (실은 브라우저 캐시)

- **증상**: 소스에서 문구를 "가중 종합 적정가"로 바꿨는데도 화면에는 여전히
  "3개 방법 평균 적정가 234,925원"이 떴다.
- **원인**: 저장소 전체를 grep해도 해당 문구는 **0건** — 소스는 이미 고쳐져 있었고,
  브라우저가 옛 `stock.js`를 캐시해서 보여주고 있었다.
- **해결**: 근본 원인은 정적 파일 캐시. 서버가 `/api/`가 아닌 정적 응답에
  `Cache-Control: no-cache`를 붙이도록 해, 수정 직후 항상 최신 JS/CSS를 받게 했다.
- **교훈**: "고쳤는데 안 바뀜"은 먼저 **소스 grep으로 사실 확인** → 캐시 의심 순으로.
  로컬 개발 서버의 정적 캐시 정책을 명시적으로 정해 두면 이런 혼선이 사라진다.

## 3. 결측 말풍선(.na / data-tip)이 머지 과정에서 유실됨

- **증상**: 빈 값(—)에 이유를 알려주는 툴팁이 어느 순간 사라져 있었다.
- **원인**: 해당 마크업은 이전 PR(커밋 `0a291f7`)에서 도입됐는데, 이후 브랜치 머지
  과정에서 되돌려졌다. 현재 트리를 grep하니 `data-tip` 0건.
- **해결**: `git show 0a291f7 -- web/stock.html web/assets/stock.js`로 원본 CSS·JS를
  찾아 재이식했다.
- **교훈**: 기능이 "사라졌다" 싶으면 `git log -S"data-tip"`(pickaxe)로 도입·삭제
  시점을 특정한다. 되살릴 코드는 항상 이력에 있다.

## 4. ADR 번호 충돌 (0003을 두 브랜치가 동시 사용)

- **증상**: 리베이스 준비 중, 내 `docs/adr/0003-peer-ranking-weights.md`와 main에 이미
  머지된 `docs/adr/0003-fair-value-weighted-average.md`가 **같은 번호**임을 발견.
- **원인**: 두 작업 브랜치가 서로 모른 채 다음 ADR 번호를 0003으로 잡았다. 파일명이
  달라 git 충돌은 안 났지만, 번호 체계가 깨진다.
- **해결**: 내 ADR을 **0005**로 재번호(0003·0004는 이미 점유)하고, 참조하는
  `README.md`·`scoring.py` docstring의 `ADR-0003`을 `ADR-0005`로 함께 수정.
- **교훈**: ADR 번호는 병렬 작업에서 충돌하기 쉽다. 리베이스 전에 대상 브랜치의
  `docs/adr/` 인덱스를 먼저 확인하고 번호를 잡는다.

## 5. 피어 병렬화 중복 — 협업자와 같은 최적화

- **증상**: 리베이스 첫 커밋(`base.py`)에서 바로 충돌. main에 이미
  `ThreadPoolExecutor`(피어 병렬 수집)가 들어와 있었다.
- **원인**: 첫 조회 지연의 주범인 순차 다운로드를, 나와 협업자가 **각자 병렬화**했다.
  구조가 달라(내 쪽은 진행 표시 훅 포함, main은 `PEER_FETCH_WORKERS` 워커 수 상수) 충돌.
- **해결**: main의 병렬 구조(중복 제거·워커 상수·상단 import)를 **채택**하고, 내 고유
  기여인 `progress.report()` 진행 표시 호출만 그 위에 이식. 중복 병렬화는 버렸다.
- **교훈**: 겹치는 최적화는 "누구 코드가 맞나"가 아니라 **이미 머지된 쪽을 기준선으로
  삼고 고유 가치만 얹는다**. 리베이스는 순수 함수 계층(진행 훅=no-op 주입)일수록 쉽다.

## 6. 헤드리스 Edge(CDP) 연결 거부로 스크린샷 실패

- **증상**: CDP 스크린샷 드라이버가 `127.0.0.1:9333` 연결 거부(WinError 10061)로 실패.
- **원인**: 이전에 띄운 헤드리스 Edge 프로세스가 종료돼 디버깅 포트가 닫혀 있었다.
  또, 백그라운드 셸 안에서 `&`로 띄운 Edge는 셸이 끝나면 함께 죽었다.
- **해결**: `cmd //c start "" msedge.exe --headless=new --remote-debugging-port=9333 ...`로
  셸과 **분리(detach)**해 기동하고, `curl .../json/version`으로 준비를 확인한 뒤 캡처.
- **교훈**: 캡처·프리뷰용 보조 프로세스는 세션 셸에 묶지 말고 detached로 띄운다.
  연결 실패 시 "포트가 살아있나"부터 확인(=헬스체크 우선).

## 7. pyflakes: 잡히지 않는 예외 변수 · 미정의 이름

- **증상**: 공개 위생 작업 중 pyflakes가 `server.py`의 `except Exception as e`(미사용 `e`)
  8건, `us_provider.py`의 `undefined name 'pd'` 1건을 지적.
- **원인**: 에러 문구를 일반화하며 예외 상세를 더 이상 쓰지 않게 됐는데 바인딩만 남음.
  provider에는 pandas 사용 코드를 추가하며 import를 빠뜨림.
- **해결**: `except Exception:`으로 정리(상세는 서버 로그로), `import pandas as pd` 추가.
  CI(Quality 워크플로)의 pyflakes가 이런 잔재를 걷어낸다.
- **교훈**: 린트는 "동작하는데 지저분한" 코드를 CI 단계에서 잡는 값싼 안전망이다.
  커밋 전 `python -m pyflakes app.py server.py src scripts tests`를 습관화.
